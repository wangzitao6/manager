# Constraints: Spring Boot 基础工程与通用能力集成

本文描述该项目在使用、开发、扩展和维护过程中需要遵守的约束条件与注意事项，避免因配置不一致、依赖冲突、敏感信息泄露或运行环境问题导致故障。

---

## 1. 技术栈约束

### 1.1 Spring Boot 版本约束
- 当前项目基于 Spring Boot 2.0.2.RELEASE
- 该版本属于较早期版本，部分新特性、Starter 兼容性和第三方库版本需要谨慎选择
- 升级 Spring Boot 版本时需同步检查：
  - Web 配置方式
  - Session 配置方式
  - Redis 客户端兼容性
  - Swagger2 兼容性
  - MyBatis-Plus 兼容性

### 1.2 Java 版本约束
- 项目使用 Java 8
- 不应直接使用高版本 Java 才支持的语法或 API
- 若升级 Java 版本，需要同步检查：
  - Lombok 兼容性
  - Maven 编译插件配置
  - 第三方 SDK 兼容性

### 1.3 MyBatis-Plus 版本约束
- 当前使用 MyBatis-Plus 2.1.9
- 该版本与较新版本的 API 存在差异
- 使用时要注意：
  - BaseMapper 的方法名和行为
  - 全局配置字段名
  - ID 策略配置
  - XML Mapper 的加载路径

## 2. 配置约束

### 2.1 服务路径约束
- 服务端口固定为 `9070`
- 上下文路径固定为 `/louxe-open-api`
- 所有前端调用和接口测试路径都必须带上上下文前缀
- 如果修改上下文路径，Swagger、拦截器、网关配置和前端请求地址都要同步调整

### 2.2 数据源约束
- 当前数据库类型为 MySQL
- 数据源使用 Druid
- 数据库连接配置必须和本地或部署环境保持一致
- `driverClassName` 当前使用的是旧版 MySQL 驱动类名 `com.mysql.jdbc.Driver`
- 若切换新版本驱动，应确认驱动名和连接参数是否兼容

### 2.3 Redis 约束
- 当前 Redis 使用 Jedis 客户端
- Redis 地址、端口和密码必须正确
- Redis 连接失败会影响缓存、Session 和相关业务流程
- 若改用其他客户端或升级 Spring Boot，需要重新检查自动配置方式

### 2.4 Session 约束
- Session 超时时间配置为 300 秒
- 这意味着用户登录态或临时状态保存时间较短
- 业务上若需要更长会话周期，必须同步调整相关配置
- 若使用 Spring Session Redis，需确保 Redis 可用，否则会影响会话管理

### 2.5 文件上传约束
- 文件最大上传大小配置为 `3000Mb`
- 请求最大大小也配置为 `3000Mb`
- 该配置较大，生产环境需要结合：
  - 磁盘空间
  - 带宽
  - Nginx 限制
  - 网关限制
  - 后端内存
  一起评估

## 3. 路径与扫描约束

### 3.1 Mapper 路径约束
- `mybatis.mapper-locations=classpath:/mapper/*.xml`
- `mybatis-plus.mapper-locations=classpath:/mapper/*Mapper.xml`
- XML 文件必须放在配置匹配的路径下，否则会出现 SQL 无法加载的问题
- Mapper 接口命名、XML 文件名和 namespace 必须保持一致

### 3.2 实体类路径约束
- 配置中写的是：
  - `mybatis-plus.typeAliasesPackage=com.wzt.mybatis.entity`
- 实际源码中的实体类包名必须与该配置保持一致
- 如果源码包名是 `com.wzt.demo.bean` 或其他路径，需要同步修正配置

### 3.3 扫描范围约束
- 启动类所在包及其子包会被 Spring Boot 默认扫描
- 如果 AOP、拦截器、配置类、Mapper、Service 不在默认扫描范围内，需要手动添加扫描配置
- 使用 `@MapperScan` 时必须确保 Mapper 包路径正确

## 4. 依赖与兼容性约束

### 4.1 Swagger2 约束
- 当前使用的是 `springfox-swagger2 2.2.2`
- 该版本较老，和某些 Spring Boot 版本存在兼容性问题
- 升级时需确认：
  - Swagger 配置类
  - 接口路径匹配规则
  - 静态资源映射
  - 文档访问地址

### 4.2 Lombok 约束
- 当前 Lombok 版本为 `1.14.4`
- 使用 Lombok 时需要 IDE 安装对应插件
- 编译环境必须支持 Lombok 注解处理
- 若升级 Lombok，需检查 getter/setter、builder、日志注解是否兼容

### 4.3 Fastjson 约束
- 项目使用的是 Fastjson
- Fastjson 在历史上存在安全风险和版本兼容问题
- 若涉及外部不可信 JSON 输入，必须注意反序列化安全
- 对敏感或复杂对象序列化时要确认输出结构是否符合预期

### 4.4 第三方 SDK 约束
- OSS 和短信服务依赖阿里云 SDK
- 使用前必须保证：
  - AccessKeyId 正确
  - AccessKeySecret 正确
  - Region 和 Domain 配置正确
  - 账号权限足够
- 密钥失效、权限不足或签名配置错误都会导致调用失败

---

## 5. 安全约束

### 5.1 敏感信息管理
- `pom.xml` 和 `application.properties` 中出现了密钥、数据库账号密码等敏感信息
- 这些信息不能直接提交到公开仓库
- 生产环境应使用：
  - 环境变量
  - 配置中心
  - 外部配置文件
  - CI/CD 注入
  来管理敏感配置

### 5.2 账号密码约束
- 当前示例中数据库用户名和密码是明文形式
- Redis 未设置密码
- OSS 和短信密钥采用占位符形式
- 正式环境必须替换为真实且受控的安全配置

### 5.3 接口安全约束
- Swagger 接口文档不应对生产环境完全开放，尤其是内部接口
- 登录接口、上传接口、短信接口等应增加权限校验和频率控制
- 对外暴露接口要考虑：
  - 参数校验
  - 重放攻击
  - 越权访问
  - 风控限制

---

## 6. 代码结构约束

### 6.1 分层约束
建议保持以下分层结构：
- Controller：接收请求
- Service：处理业务
- Mapper：数据库访问
- Entity/Bean：数据对象
- Config：配置类
- Interceptor：拦截器
- AOP：切面增强
- ThirdPart：第三方服务封装
- Utils：工具类

### 6.2 包职责约束
- `bean` 包只放数据对象，不要混入复杂业务逻辑
- `utils` 包只放无状态工具类
- `thirdpart` 包只负责第三方 API 封装
- `config` 包只负责配置类，不要写业务逻辑
- `cache` 包只做缓存封装，不要直接耦合 Controller

### 6.3 XML 与 Java 代码约束
- XML Mapper 中的 SQL 要与接口方法参数和返回值保持一致
- namespace 必须和 Mapper 接口全限定名一致
- resultType、parameterType、id 等字段必须正确

---

## 7. 运行环境约束

### 7.1 本地环境约束
- 本地必须具备：
  - JDK 8
  - Maven
  - MySQL
  - Redis
  - 可访问的 OSS / 短信配置或模拟环境
- 本地数据库、Redis 地址和密码要与配置匹配

### 7.2 部署环境约束
- 部署前要确认：
  - 端口未被占用
  - 数据库连接可用
  - Redis 连通正常
  - 文件上传目录或 OSS 配置可用
  - 网关和反向代理未拦截大文件上传请求

### 7.3 网络环境约束
- 阿里云 OSS 和短信服务依赖公网访问
- 若服务器在内网环境，必须保证有外网出口
- Redis 使用远程地址时也要确保网络可达和防火墙放行

---

## 8. 功能使用约束

### 8.1 缓存使用约束
- 缓存数据必须考虑过期时间
- 不应把高一致性强依赖的数据长期只放 Redis
- 缓存失效时要有回源机制
- 注意缓存穿透、缓存击穿、缓存雪崩问题

### 8.2 Session 使用约束
- Session 适合保存短期登录态或临时状态
- 不建议在 Session 中存放大对象
- 分布式部署下必须确保 Session 统一存储到 Redis

### 8.3 上传使用约束
- 上传文件要校验大小、格式和后缀
- 若直传 OSS，需要处理上传失败和重试
- 若落本地磁盘，需要考虑清理策略和磁盘占满风险

### 8.4 短信使用约束
- 短信验证码需限制发送频率
- 验证码应有过期时间和校验次数限制
- 不应将短信接口作为高频无保护开放接口

---

## 9. 调试与测试约束

### 9.1 测试环境约束
- 测试库、测试 Redis、测试 OSS 配置应与生产隔离
- 不能直接在生产数据库上进行开发调试
- 使用 profile 或外部配置文件区分环境

### 9.2 单元测试约束
- 使用 `spring-boot-starter-test`
- 单元测试应尽量避免依赖真实外部服务
- 对 OSS、短信、Redis 等外部依赖建议进行 mock 或隔离测试

### 9.3 日志约束
- 日志应输出关键流程、异常和耗时
- 不建议把密码、密钥、验证码等敏感信息写入日志
- 生产环境日志级别应合理控制，避免过多 debug 输出

---

## 10. 维护约束

### 10.1 版本升级约束
- 该项目依赖中存在多个较旧版本
- 升级依赖时必须逐项验证兼容性
- 重点关注：
  - Spring Boot
  - MyBatis-Plus
  - Swagger
  - Redis 客户端
  - Fastjson
  - Lombok

### 10.2 重构约束
- 若重构包名或模块结构，必须同步修改：
  - `typeAliasesPackage`
  - `mapper-locations`
  - `@MapperScan`
  - `contextPath`
  - 拦截器路径
  - Swagger 配置

### 10.3 配置同步约束
- `pom.xml`
- `application.properties`
- 源码包结构
- README
- SQL 初始化脚本

这几部分必须保持一致，否则容易出现“代码和配置不匹配”的问题。

---

## 11. 明确不应做的事情
- 不要把真实密钥直接提交到仓库
- 不要把 MongoDB 当成已启用功能写进说明
- 不要假设实体类路径一定与配置一致，必须以源码确认
- 不要在生产环境直接使用开发环境密码和 Redis 配置
- 不要忽略 Swagger 和拦截器、上下文路径之间的联动关系
- 不要在未验证兼容性的情况下升级核心依赖

---

## 12. 总结
该项目是一个典型的 Spring Boot 基础工程模板，能力较全，但也存在早期版本依赖较老、配置较分散、敏感信息管理不够规范等问题。

在使用和扩展时，必须重点关注以下几点：
- 保持包路径和配置一致
- 保护敏感信息
- 控制依赖版本兼容性
- 管理好 Redis、Session、MySQL 和第三方服务
- 避免把未启用功能写成已启用能力

只有在这些约束下，项目才适合作为稳定的基础模板继续演化。
