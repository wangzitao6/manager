- `src/main/java/com/wzt/demo/cache`

## Rules

### 1. 不要默认 Redis 已启用
如果项目代码和配置中没有明确 Redis 相关内容，不要直接假设该功能可用。

### 2. Key 命名必须统一
建议统一使用可读、可维护的命名方式，例如：

- `module:biz:id`
- `user:profile:123`
- `sms:code:138xxxx`
- `login:token:xxx`

### 3. TTL 必须与业务场景匹配
不同缓存类型应有不同过期时间，例如：

- 验证码类：短 TTL
- 登录态：中等 TTL
- 热点数据：较长 TTL
- 临时任务状态：按业务流程设置

### 4. 优先保证一致性策略清晰
写缓存前要考虑：

- 先写数据库再删缓存
- 先删缓存再写数据库
- 双删策略
- 延迟双删
- 失效重建

### 5. 缓存穿透、击穿、雪崩必须可识别
当业务场景涉及高并发或热点查询时，应主动判断是否需要：

- 空值缓存
- 布隆过滤器
- 互斥锁
- 随机过期时间
- 限流

### 6. 序列化方式要明确
必须确认缓存对象采用何种序列化方式，避免：

- 反序列化失败
- 数据乱码
- 兼容性问题

### 7. 不要把所有数据都缓存
缓存应优先用于：

- 读多写少
- 查询代价高
- 变化不频繁
- 允许短时间不一致的数据

## Steps

### Step 1: 识别缓存使用场景

判断当前场景属于以下哪类：

- 普通查询缓存
- 验证码缓存
- 会话缓存
- 热点数据缓存
- 业务状态缓存
- 分布式锁
- 限流
- 去重标记

---

### Step 2: 设计 Key 结构

为不同业务设计统一命名空间，例如：

- `user:info:{id}`
- `order:detail:{id}`
- `sms:code:{phone}`
- `login:token:{token}`

Key 命名应满足：

- 可读
- 可扩展
- 不冲突
- 易排查

---

### Step 3: 设计 TTL 策略

根据业务特征设置过期时间：

- 临时验证码：短 TTL
- 用户资料：中长 TTL
- 配置类数据：长 TTL
- 状态标记：按流程设置 TTL

若需要随机过期，应避免同一时刻大量失效。

---

### Step 4: 设计缓存读写流程

常见模式包括：

- 先查缓存，未命中再查数据库
- 查库后回写缓存
- 更新数据库后删除缓存
- 批量预热缓存
- 异步刷新缓存

应根据数据特性选择最合适的模式。

---

### Step 5: 处理高并发问题

识别并防护以下问题：

- 缓存穿透
- 缓存击穿
- 缓存雪崩
- 热点 Key
- 缓存一致性

必要时引入：

- 空对象缓存
- 分布式锁
- 本地锁
- 逻辑过期
- 双删策略

---

### Step 6: 检查序列化与反序列化

确认缓存对象是否需要：

- JSON 序列化
- JDK 序列化
- 自定义序列化
- String 存储

若对象复杂，建议优先使用结构清晰、可调试的序列化方式。

---

### Step 7: 输出封装建议

如果项目已有缓存封装，应检查是否需要统一：

- `CacheService`
- `RedisKeyConstants`
- `CachePrefix`
- `RedisUtil`

若没有封装，可建议补充统一工具类。

## Examples

### Example 1: 用户信息缓存

输入：

```text
用户信息查询频繁，但修改不频繁
```

输出方向：

- Key：`user:info:{userId}`
- TTL：中等
- 策略：先查缓存，未命中查库并回写

---

### Example 2: 验证码缓存

输入：

```text
短信验证码需要短时间有效
```

输出方向：

- Key：`sms:code:{phone}`
- TTL：短
- 策略：写入后仅用于校验一次或多次，视业务而定

---

### Example 3: 登录态缓存

输入：

```text
用户登录后保存 token
```

输出方向：

- Key：`login:token:{token}`
- TTL：会话型
- 策略：登录成功时写入，退出时删除，过期自动失效

---

### Example 4: 防缓存穿透

输入：

```text
大量不存在的 ID 被频繁查询
```

输出方向：

- 对空结果进行短期缓存
- 可结合布隆过滤器
- 对非法参数进行前置校验

## Pitfalls

### 1. Key 命名混乱
如果不同模块随意命名 Key，会导致：

- 冲突
- 难排查
- 失效策略不统一

### 2. TTL 设置不合理
TTL 过短会导致频繁回源数据库，TTL 过长则可能造成数据不一致。

### 3. 更新缓存与更新数据库顺序错误
如果写入顺序设计不合理，容易出现脏数据或旧值覆盖新值。

### 4. 忽略热点 Key 问题
热点 Key 容易在高并发下成为瓶颈，应特别注意限流、互斥锁和逻辑过期。

### 5. 序列化不一致
不同代码路径使用不同序列化方式，会导致读取失败或数据解析错误。

### 6. 把缓存当数据库
Redis 适合做缓存和临时存储，不适合盲目作为核心持久化数据的唯一来源。
